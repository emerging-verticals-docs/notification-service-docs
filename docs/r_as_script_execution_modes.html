Script Execution Modes

You can configure a script to run in the following modes: pre-processing, post-processing, or custom.

Pre-processing scripts

A script with @execution pre-processing runs before the scripting engine forwards a request to the Asset service. The script can modify the body of the request or cancel the request altogether. A pre-processing script can also issue GET requests to the Asset service if more information is needed to process the request.

Post-processing scripts

A script with @execution post-processing runs after a request has been processed by the Asset service and had its response sent to the client. A post-processing script can issue additional GET, PUT, POST, DELETE, and PATCH requests to the Asset service. Since post-processing scripts are executed after a response has been sent to the client, there is no way to alert the client directly should a failure occur. Instead, the Scripting Engine creates a system message containing the reason for the failure. Clients should actively monitor their system messages for post-processing failures.

Custom-endpoint scripts

A script with @execution custom runs when a request is sent to a custom endpoint (such as /custom/my-logic). A custom-endpoint script can issue GET, PUT, POST, DELETE, and PATCH requests to the Asset service. Custom-endpoint scripts can send a custom response back to the client.
